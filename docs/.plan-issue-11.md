# Execution plan — issue #11 E2E test coverage hardening

## Intent

Implement issue #11 as a layered E2E hardening effort that:

- keeps real LinkedIn actions safe and explicit
- gives CI a truthful skip-or-run contract
- expands from core-runtime coverage to CLI and MCP surfaces
- fills the missing two-phase commit and error-path coverage without making the
  default contributor workflow risky

## Recommended approach

### 1) Centralize E2E gating before adding more tests

The first step should be to replace the current per-test early-return pattern in
`packages/core/src/__tests__/e2e/setup.ts` with one explicit E2E harness.

That harness should:

- probe CDP reachability once
- probe authentication once
- report a clear "executed" versus "skipped" state
- make missing-session skips visible in test output
- use the existing `LINKEDIN_CDP_URL` env var consistently
- either adopt or remove the currently unused `LINKEDIN_E2E` env var contract

This is the prerequisite for trustworthy CI behavior.

### 2) Split the live suite by safety level

The current E2E directory mixes read-only smoke tests with real outbound and
public actions. Issue #11 should separate those into explicit lanes.

Recommended lane model:

- **Read-only lane**
  - auth status / health
  - profile view
  - search
  - jobs search / view
  - inbox list / show
  - connections list / pending
  - feed list / view
  - notifications
  - selector audit
- **Private write lane**
  - message send
  - connection invite / accept / withdraw
  - follow-up after accept
- **Public write lane**
  - like
  - comment
  - post create

The default E2E command should only ever run the read-only lane automatically.
Write lanes should require explicit opt-in flags.

### 3) Fill the core-runtime coverage gaps first

Before adding CLI and MCP wrappers, complete the missing core live coverage.

Suggested additions:

- `inbox.getThread()`
- `feed.viewPost()`
- `jobs.viewJob()`
- `connections.listPendingInvitations()`
- `feed.prepareLikePost()` prepare → confirm
- `connections.prepareSendInvitation()` prepare → confirm
- `connections.prepareAcceptInvitation()` prepare → confirm
- `connections.prepareWithdrawInvitation()` prepare → confirm
- `followups.listAcceptedConnections()`
- `followups.prepareFollowupsAfterAccept()` and confirm path where safe

This keeps the LinkedIn interaction logic validated in one place before adding
process-level wrappers.

### 4) Add dedicated error-path E2E coverage

Issue #11's error-path requirements should be treated as first-class suites, not
as incidental assertions inside happy-path tests.

Recommended categories:

- **Expired token**
  - use a short-lived prepared action or controllable expiry in a dedicated test
  - confirm after expiry and assert the structured failure
- **Rate limited**
  - use controlled limiter state where possible
  - assert confirm-time failure behavior, not only preview metadata
- **UI change detection**
  - validate `UI_CHANGED_SELECTOR_FAILED` handling through selector audit and/or
    a controlled selector-failure harness
  - assert artifact expectations for diagnosis, not just error codes

### 5) Add thin CLI E2E wrappers after core coverage is solid

Once the runtime surface is covered, add CLI process tests that execute the real
binary and assert the operator contract.

The CLI layer should validate:

- argument parsing
- JSON output shape
- exit codes
- alias behavior such as `feed like` / `feed react`
- `actions confirm` and `post confirm` flows
- CLI-only operations like `rate-limit`, `keepalive`, and `audit selectors`

These tests should stay thin: they should reuse the same targets and expectations
already proven at the runtime layer instead of re-encoding LinkedIn behavior.

### 6) Add MCP stdio integration coverage after CLI coverage

Issue #11 also requires MCP tool coverage, but the MCP server should be tested
as a transport and contract layer, not as a second implementation of LinkedIn
logic.

Recommended scope:

- start the real `linkedin-mcp` stdio server
- enumerate or directly invoke registered tools
- validate request/response payload shape
- validate structured error behavior
- reuse the same fixture targets already defined for runtime and CLI tests

Before doing this, resolve the current inventory mismatch between
`LINKEDIN_SESSION_HEALTH_TOOL` in `packages/mcp/src/index.ts` and the absence of
a registered health handler in `packages/mcp/src/bin/linkedin-mcp.ts`.

### 7) Wire a CI-compatible runner last

After the harness and suites are split correctly, add the CI-compatible runner.

That runner should:

- execute read-only E2E when a session is available
- emit explicit skipped output when no authenticated session is available
- leave hosted GitHub Actions green when session prerequisites are absent
- optionally fail in stricter environments when a session is expected but missing
- keep private/public write suites disabled by default

This gives the repo one portable E2E entry point that works locally, in managed
agent sessions, and in CI.

## Suggested files to touch, in order

1. `packages/core/src/__tests__/e2e/setup.ts`
   - centralize probe and skip logic
   - expose suite/lane helpers instead of raw booleans

2. `packages/core/src/__tests__/e2e/`
   - split tests into read-only, private-write, and public-write groups
   - add the missing runtime coverage and error-path suites

3. `scripts/`
   - add one E2E runner script if Vitest alone is not enough to express skip
     policy and summary output

4. `package.json`
   - expose clearer E2E script entry points for read-only versus opt-in writes

5. `packages/cli/src/bin/linkedin.ts`
   - no behavior changes unless tests expose a real operator-surface bug

6. `packages/mcp/src/bin/linkedin-mcp.ts`
   - only fix true server-surface gaps needed for deterministic MCP tests

7. `packages/mcp/src/index.ts`
   - reconcile tool inventory with registered handlers

8. `README.md`
   - replace the current broad E2E section with lane-based guidance and explicit
     safe-target rules

9. `docs/`
   - add one operator-facing E2E guide that centralizes prerequisites,
     safe-target policy, and CI skip behavior

## Main risks and blockers

### 1) Safe target state for connection flows

Connection invite / accept / withdraw / follow-up tests need controlled LinkedIn
state, not just a friendly target name. A dedicated fixture-account strategy may
be required before those flows can be fully automated.

### 2) Public-action risk

Comments, likes, and posts affect real LinkedIn surfaces. They should never be
part of the default E2E lane and must remain explicitly approved and opt-in.

### 3) False-green CI results

If the project keeps the current early-return pattern, a future CI E2E job could
look healthy without having executed anything. That needs to be fixed before the
coverage matrix expands.

### 4) Duplicate test logic across core, CLI, and MCP

The runtime layer should remain the source of truth for LinkedIn behavior. CLI
and MCP tests should verify contract and transport behavior, not duplicate all
interaction logic from scratch.

### 5) Hosted CI reality

The existing GitHub Actions workflow does not provision a real authenticated
LinkedIn session. Issue #11 should therefore target a runner that can skip
truthfully on hosted CI and execute fully only in environments where a session
is intentionally provided.

## Guardrails

### Do touch

- E2E harness and test organization
- runtime E2E gaps required by issue #11
- thin CLI and MCP test harnesses
- E2E documentation and safety guidance
- CI runner entry points and skip semantics

### Do not default to

- running public write flows automatically
- hiding missing-session skips behind silent returns
- forcing hosted CI to provision real LinkedIn credentials
- duplicating LinkedIn interaction logic in CLI and MCP tests

## Implementation compass

If issue #11 stays focused on these three questions, scope should stay healthy:

1. Did a real test actually execute, or was it explicitly skipped?
2. Is the surface under test the runtime, the CLI contract, or the MCP contract?
3. Is the action read-only, private-write, or public-write?

If every new test and script answers those three questions clearly, the project
can harden E2E coverage without making the repo unsafe or misleading.
