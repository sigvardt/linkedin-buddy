# Execution plan — issue #88 E2E live smoke tests: read-only validation CLI (Tier 2)

## Intent

Implement issue #88 as a **live, operator-facing read-only validation lane**
that runs against a real authenticated LinkedIn session and proves that a small,
high-value set of browser flows still works.

This should **extend the current runtime, CLI, and E2E harness** rather than
introducing a second Playwright stack, a new session model, or a broad base-URL
rewrite. Tier 1 fixture replay from issue #87 is already the right foundation
for deterministic CI coverage; Tier 2 should sit on top of that foundation as
the **live drift-detection and validation layer**.

Phase-1 scope should stay deliberately narrow:

- keep `npm run test:e2e` as the existing live-session runner
- add a dedicated CLI surface for live read-only validation
- reuse selector-audit-style reporting, artifacts, and exit semantics
- prove a curated safe check set before broadening to every read-only feature
- build reusable live-execution policy primitives that issue #90 can later
  tighten for write validation

Treat `docs/.research-brief-issue-88.md`, `docs/.plan-issue-87.md`, and
`docs/e2e-testing.md` as the planning baseline. When the issue wording and the
current codebase differ, prefer the least-invasive shape that fits the existing
profile-based session model and keeps the default lane read-only.

## Recommended approach

### 1) Add a dedicated live-validation command group

Add an operator-facing command on the existing CLI surface:

- `linkedin test live --read-only`
- hidden alias: `linkedin test:live --read-only`
- naturally available as `buddy ...` through the existing bin alias

This should be a **new top-level command group**, not a pile of ad hoc flags on
unrelated commands. Follow the same hidden colon-alias pattern already used by
`fixtures`.

The command contract should stay small and predictable:

- `--profile` to choose the persistent Playwright profile
- `--read-only` as an explicit safety flag, even if it is the only supported
  mode in phase 1
- `--yes` to bypass an optional preflight confirmation
- `--json` for automation
- optional persisted report output if the implementation needs one, but only if
  it reuses the existing artifact/report plumbing

For confirmation UX, prefer **one run-level confirmation of the generated plan**
instead of a prompt before every step. Tier 2 is valuable because it is safe to
run in batch mode; do not overbuild prompt choreography unless the issue owner
explicitly insists on step-by-step approval.

### 2) Put orchestration and safety policy in `packages/core`

The main missing abstraction is not browser automation. It is a **shared live
validation policy layer** that can enforce safety and pacing regardless of how
the CLI is wired.

The core implementation should likely introduce one small module responsible
for:

- defining the live-validation step contract
- building the ordered run plan for the read-only lane
- enforcing request pacing and a per-run request budget
- refusing non-GET LinkedIn traffic in read-only mode
- aborting on login redirects, checkpoint/challenge URLs, or other session
  anomalies
- collecting per-step outcomes, artifacts, timings, and report metadata

This policy should live in `packages/core`, not inline in the CLI, because
issue #90 will almost certainly need the same primitives with stricter rules.

Keep the first extraction small. A typed orchestrator plus a typed report is
enough. Do **not** start with a generic plugin framework.

### 3) Reuse selector audit as the architectural template

Selector audit is the closest existing model because it already:

- performs one authenticated run
- uses one context across multiple checks
- emits a structured report
- captures artifacts on failure
- exposes clear pass/fail behavior on the CLI

Tier 2 should follow the same shape:

1. perform one upfront auth/health preflight
2. build the live read-only run plan
3. execute checks through one shared context where possible
4. collect one report and one terminal summary
5. exit non-zero on validation failure or regression

This matters because it avoids the biggest likely Tier-2 failure mode:
stringing together many existing commands and paying repeated auth/navigation
costs for each one.

### 4) Use Tier 1 as the foundation, not as a competing lane

Issue #87 already established the repo’s test-lane shape:

- deterministic replay lane for CI-safe regression coverage
- live runner with explicit availability/skip behavior
- small discovery-fixture helpers for live IDs used by thin CLI/MCP tests
- one shared operator document for E2E workflow and safety

Tier 2 should **reuse that shape**:

- keep replay and live semantics clearly separate
- reuse the existing live E2E runner/help/update patterns instead of inventing
  a second runner philosophy
- reuse the fixture/discovery mindset for optional live identifiers such as an
  inbox thread used by a read-only detail step
- extend `docs/e2e-testing.md` rather than creating a disconnected workflow doc

Tier 1 remains the regression floor. Tier 2 adds a live smoke layer on top of
it; it should not weaken or blur the replay story.

### 5) Keep the first live step set narrow and safe

The first implementation should prove a small set of read-only checks that are
already valuable:

- session health / authentication preflight
- selector audit
- feed list
- profile view for `me`
- connections list
- inbox list
- inbox thread view using a discovered or cached safe thread identifier when
  available

`notifications list` is also a good target **if** its context handling can be
harmonized cheaply with the shared live-validation flow. If that harmonization
starts to sprawl, defer notifications rather than broadening the first slice
with uneven context behavior.

`search`, `jobs`, and `followups` should not be first-slice requirements:

- `search` and `jobs` currently have the clearest shared-context / CDP mismatch
- `followups list` is read-only on LinkedIn but writes acceptance state to the
  local DB, which makes it a poor fit for a “strictly read-only validation” UX

### 6) Make reports and artifacts first-class from the start

Do not invent a second reporting stack. Reuse the repo’s existing JSON/logging
and artifact conventions.

The Tier-2 report should capture at least:

- run metadata: profile, timestamp, selector locale, read-only mode, pacing
  policy, request budget
- per-step status: pass, fail, skip, soft-fail, aborted
- step timings and lightweight result summaries
- artifact paths for any captured screenshots or failure details
- failure classification that distinguishes auth problems, policy violations,
  selector drift, and ordinary step errors

Design the report schema so a baseline/diff helper can be added without a
rewrite. Stable step identifiers and explicit outcomes matter more than a fancy
diff engine in phase 1.

### 7) Harmonize context usage cautiously

The biggest structural risk in the brief is that some read-only services honor
the runtime’s shared context pathway while others still go directly to
`runWithPersistentContext()`.

Treat this as a **small extraction problem**, not as a mandate to refactor the
entire service layer.

Good direction:

- introduce one tiny shared seam for live-validation context/policy injection
- migrate only the services that Tier 2 truly needs in the first slice
- if a service still resists the shared-context model, leave it out of phase 1

Bad direction:

- broad “replace every `runWithPersistentContext()` call” refactors
- mixing live-validation policy concerns into unrelated feature code

## Suggested implementation slices

### Slice 1: Core contracts and policy

Define the typed run contract first:

- live-validation step definition and ordered plan
- live-validation report schema
- read-only policy object with pacing, request budget, and abort rules
- a small error taxonomy for policy violations versus ordinary check failures

This is the slice that makes the rest of the work additive instead of ad hoc.

### Slice 2: Core orchestrator and first step set

Implement the core service that:

- runs the preflight auth/health check once
- executes the initial safe step set
- shares context where possible
- records outcomes, artifacts, and exit-worthy failures

Start with the checks that already fit the current architecture best. Avoid
inflating phase 1 with every read-only surface just because the methods exist.

### Slice 3: CLI wiring and output

Add the `test live` / `test:live` commands on the existing binary, keeping the
CLI thin:

- parse flags
- print the preflight plan/confirmation prompt when appropriate
- call the core orchestrator
- render human or JSON output
- set exit codes from report outcomes

If the human-readable formatting grows beyond a few inline lines, introduce a
small sibling formatter module rather than expanding `linkedin.ts` further.

### Slice 4: Test coverage

Add tests at the narrowest useful layer first:

- unit tests for policy decisions, report shaping, and fail-closed behavior
- CLI tests for argument parsing, confirmation, JSON/human output, and exit
  codes
- live E2E coverage through the existing runner and shared helpers

The live E2E slice should prove the command contract and the safe default lane.
It should not try to simulate every possible LinkedIn drift mode in one pass.

### Slice 5: Documentation and operator workflow

Update the existing E2E guide after the command exists:

- what the Tier-2 lane validates
- how it differs from fixture replay
- required session prerequisites
- read-only safety guarantees and current exclusions
- how to interpret failures, skips, and artifact output

## New code that is justified

The first implementation will likely need some net-new code, and that is
appropriate here:

- one small core live-validation orchestrator
- one typed live-validation policy object/helper
- one typed report shape and optional diff helper
- one thin CLI command group for `test live`
- focused unit tests for fail-closed policy and report behavior
- focused live E2E coverage for the new command contract
- a small shared-context extraction only if the chosen first-slice checks truly
  require it

That is enough for issue #88’s first milestone. Anything larger should clear a
high scope bar.

## Main risks to watch

### 1) Shared-context support is inconsistent across read-only services

Some read-only services already fit the shared-context model better than others.
If Tier 2 starts by demanding full parity across every module, the work can
quickly turn into a broad refactor.

Planning response:

- start with the checks that already align well
- extract one small shared-context seam where it pays off
- defer outlier services instead of broadening the refactor

### 2) Repeated auth/status checks will add traffic and latency

Naively chaining existing commands can trigger repeated authenticated
navigations.

Planning response:

- do one preflight auth/health check
- reuse one shared context where possible
- model the command after selector audit, not after a shell script that calls
  many commands serially

### 3) Inbox is safe but operationally flakier than other read-only checks

Inbox relies on more network-sensitive behavior than profile or feed listing.

Planning response:

- treat inbox detail as a soft or later step if needed
- let the report distinguish skip/soft-fail from hard policy violations
- do not let inbox fragility block the entire phase-1 lane design

### 4) “Read-only on LinkedIn” is not always “read-only in the repo”

`followups list` updates local state, which makes it a poor first-slice fit for
strict read-only validation.

Planning response:

- exclude followups from phase 1
- keep the read-only promise strict and easy to explain

### 5) Session-capture wording can trigger UX scope creep

The issue language talks about captured sessions, but the product already uses a
profile-based persistent Playwright session model.

Planning response:

- build Tier 2 on top of the current `login --profile` flow
- treat `auth:session` UX as follow-on work unless explicitly required

### 6) Locale handling can quietly invalidate the signal

Tier 2 exists partly to detect real-site selector drift. Locale-blind reports
will muddy that signal.

Planning response:

- carry selector locale into report metadata
- reuse existing locale-aware selector plumbing
- avoid English-only assumptions in step expectations or summaries

### 7) A non-fail-closed “read-only” lane can create false confidence

If a future code path leaks a mutation request, best-effort logging is not good
enough.

Planning response:

- make the live-validation policy fail closed on disallowed traffic
- treat policy violations as immediate aborts, not soft warnings
- test that behavior directly at the unit layer

### 8) Scope can drift into Tier 3 too early

Issue #90 will need overlapping infrastructure, but Tier 2 should not become a
half-built write-validation system.

Planning response:

- share policy/report primitives
- keep the step set read-only
- avoid wiring mutation helpers or confirm flows into the Tier-2 command

## Guardrails

### Do touch

- a new core live-validation module and its typed helpers
- runtime wiring for that module
- the CLI command group for `test live`
- unit tests and live E2E coverage for the new command
- the existing E2E guide and any closely related help text
- small shared-context extractions needed by the chosen first-slice checks

### Do not touch unless the implementation proves it is necessary

- headless email/password login flows in `packages/core/src/auth/session.ts`
- two-phase commit behavior and confirm flows
- the replay manifest/storage architecture from issue #87
- MCP surfaces in the first slice; issue #88 asks for a CLI, not a second
  transport rollout
- broad `linkedin.com` URL rewrites across feature modules
- new test frameworks, browser stacks, or heavy runtime dependencies

### Avoid these traps

- do not model Tier 2 as a shell script that calls many existing commands one by
  one
- do not weaken the read-only promise with “best effort” mutation blocking
- do not include `followups` just because it looks read-only at first glance
- do not broaden the first slice with `search`/`jobs` just for feature-count
  parity while shared-context semantics are unresolved
- do not let live and replay workflows drift into separate, conflicting docs
- do not bury checkpoint/challenge failures inside generic step errors
- do not overbuild interactive confirmation for a lane whose main value is safe
  batch validation

## Conventions to follow

- keep TypeScript strict and additive
- use ESM imports with `.js` extensions
- use `node:` imports for built-ins
- use type-only imports where appropriate
- keep the CLI thin; put orchestration and policy in core
- prefer one preflight auth check and one shared context where possible
- preserve structured JSON logging and artifact capture
- prefer small typed helpers over a generalized step/plugin framework
- keep human output TTY-friendly and JSON output automation-friendly
- include selector locale and safety-policy metadata in the report
- keep live and replay lanes explicitly separate in both code and docs

## Implementation compass

If the implementation lands as a small core orchestrator, a thin `test live`
CLI wrapper, a narrow safe step set, one selector-audit-style report, focused
unit coverage, and a live E2E suite that reuses the current runner/helpers, it
is probably on track.

If it starts introducing a new session model, broad feature-service refactors,
per-feature bespoke reports, write-flow plumbing, or fuzzy “read-only unless we
miss something” semantics, it has probably drifted out of scope.
