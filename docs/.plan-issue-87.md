# Execution plan — issue #87 E2E test harness: recorded fixture replay (Tier 1)

## Intent

Implement issue #87 as a **new, deterministic fixture-backed E2E lane** that replays recorded LinkedIn browser traffic and page state through Playwright so the repo can run high-value browser coverage in CI **without a live authenticated LinkedIn session**.

The implementation should **extend the existing Vitest + Playwright-library harness** instead of replacing it with a second test stack or a clean-room mock architecture. The current live E2E lane, shared suite setup, artifact capture, selector audit, and small CLI/MCP fixture file are already the right foundation.

Phase-1 scope should stay deliberately narrow:

- keep `npm run test:e2e` as the live-session lane
- add a separate fixture-backed lane for CI-safe replay
- prioritize read flows, selector audit, and prepare/preview flows before broadening replayed confirm coverage
- keep live login, keepalive, and live write validation out of scope
- treat fixture capture and sanitization as an operator workflow, not a production runtime concern

Treat `docs/.research-brief-issue-87.md`, issue #87, and issue #111 as the source of truth. When the issue wording and the current codebase pull in different directions, prefer the least-invasive harness seam that still guarantees “no real LinkedIn traffic.”

## Recommended approach

### 1) Add a fixture-backed E2E lane, not a replacement for the live lane

Keep the current live runner and skip semantics intact. Add a separate replay lane with its own runner and/or Vitest config so CI can run fixture-backed browser tests without CDP, saved cookies, or operator approval.

This keeps the existing developer workflow clear:

- live session lane for real drift checks and approved writes
- replay lane for deterministic regression coverage
- unit tests for harness plumbing and edge cases

### 2) Put replay behind a dedicated browser-context adapter

The core services already navigate to real LinkedIn URLs. Phase 1 should take advantage of that by introducing one replay adapter that creates an ephemeral Playwright context for record/replay and keeps feature modules unchanged wherever possible.

That adapter should own concerns such as:

- HAR recording and replay
- service-worker blocking
- fail-closed behavior for missing routes
- fixture metadata and capture directories
- any optional HTML snapshot attachment needed for debugging

If a literal localhost mock server is still required to satisfy the parent issue, keep it behind the same adapter and do not start by rewriting every hardcoded LinkedIn URL.

### 3) Make auth deterministic before broadening coverage

Most services call `runtime.auth.ensureAuthenticated()` before they do useful work. The replay lane therefore needs a deterministic auth story very early.

A good phase-1 direction is to keep the live auth implementation intact and add a replay-side seam that can do one of these cleanly:

- infer authenticated state from replay fixtures and a known feed/login surface
- or provide a test-only authenticated auth adapter inside the replay runtime

The important part is to solve auth once at the harness boundary, not ad hoc inside each feature suite.

### 4) Treat fixtures as manifest-backed capture sets, not one-off files

The repo already has a versioned, profile-aware JSON fixture file for CLI/MCP stable IDs. Reuse that mindset for Tier 1, but scale it up to page fixtures.

The replay corpus should distinguish between:

- a committed manifest with format/version/metadata
- large HAR and HTML assets that stay gitignored by default
- capture sets partitioned by page type, locale, and account state
- freshness and sanitization status that operators can inspect and refresh

This is also the right place to support incremental rerecording of only the stale or broken pages.

### 5) Start with the best proof points, not full parity on day one

The harness will be more valuable if it proves the hardest paths early instead of chasing breadth too fast.

A sensible phase-1 order is:

1. replay runtime/context + manifest
2. feed/profile/notifications/search/jobs read coverage
3. selector audit replay suite on high-value pages
4. inbox replay proof point, including `waitForResponse()` behavior
5. connections, post composer, and followups with explicit account-state fixture sets
6. replayed confirm-path coverage only after prepare/preview paths are stable

This still leaves room for the builder to choose the exact first test files.

### 6) Keep fixture capture and validation on the existing CLI surface

Issue #87 calls for `fixtures:record` and `fixtures:check`. In this repo, those commands should live on the existing `linkedin` CLI rather than inventing a separate tool surface.

The CLI should stay thin:

- parse operator inputs
- call core/test harness helpers
- print useful manifest/check output
- avoid embedding browser orchestration or fixture parsing inline

### 7) Add CI as a first-class fixture lane

The replay lane should land with an explicit CI story instead of assuming current `npm test` is enough.

That probably means:

- a dedicated script for fixture-backed E2E
- browser provisioning in CI for that job
- a clear separation from the current fast unit/integration job
- failure output that still preserves current artifact/debug habits

## Suggested implementation slices

### Slice 1: Fixture manifest and storage conventions

Choose the fixture directory layout, manifest schema, format versioning, freshness policy, and gitignore rules first. This gives the rest of the work a stable contract.

### Slice 2: Replay context/runtime seam

Add the replay browser-context factory and the auth/runtime seam so tests can execute against fixtures without touching the live CDP path.

### Slice 3: First replay suites

Bring up a small set of high-value read suites plus selector audit. Use them to validate offline replay, artifact collection, and suite ergonomics before broadening coverage.

### Slice 4: Inbox and stateful fixture sets

Add the first network-aware replay coverage and at least one state-sensitive surface so the harness proves it can handle more than static page extraction.

### Slice 5: Capture/check CLI, CI, and docs

Add operator capture/refresh/check commands, wire the new lane into CI, and document the workflow once the implementation shape is settled.

## Suggested files to touch, in order

This order is meant to keep the work incremental and reduce churn.

1. `.gitignore`

   - keep fixture storage outside `src` and away from normal build inputs

   - isolate replay suites from live-session suites
   - keep the new lane obvious in local and CI workflows

   - reuse shared flag/help parsing only where it actually reduces duplication
   - avoid overloading the live-session runner with two different execution models

   - add a replay-aware suite setup path or extract shared setup pieces
   - keep current availability/skip helpers intact for live tests

6. `packages/core/src/__tests__/e2e/helpers.ts`
   - extend versioned fixture helpers and manifest parsing
   - keep the small CLI/MCP stable-ID fixture flow distinct from full page replay assets

7. `packages/core/test/fixtures/` or another dedicated test-fixture directory
   - add the manifest and storage layout for HAR/HTML capture sets
   - keep large assets out of git unless there is a deliberate exception

8. `packages/core/src/profileManager.ts` and/or one new replay harness module in `packages/core/src/`
   - add ephemeral record/replay context creation
   - block service workers and fail closed on missing fixture routes

9. `packages/core/src/runtime.ts`
   - wire replay-specific runtime/auth seams only as far as needed to avoid duplication
   - keep production runtime defaults live-session-safe

10. `packages/core/src/auth/session.ts`
    - touch only if the replay lane truly needs a minimal authenticated seam here
    - keep real login and challenge handling out of phase 1

11. `packages/cli/src/bin/linkedin.ts`
    - add `fixtures record` and `fixtures check` commands on the existing CLI
    - reuse the repo’s thin CLI-to-core handoff style

12. `packages/core/src/__tests__/e2e/*.e2e.test.ts`
    - add or split fixture-backed suites for the prioritized modules
    - keep the first slice small enough that failures are easy to diagnose

13. `.github/workflows/ci.yml`
    - add browser provisioning and a dedicated fixture E2E job
    - keep the current fast job fast

14. `docs/e2e-testing.md` and, if needed, `README.md`
    - document capture, refresh, sanitization, and CI workflow after the harness exists
    - prefer one clear workflow doc over scattered partial instructions

## New code that is justified

The first implementation will likely need some net-new code, and that is appropriate here:

- one replay harness or context-factory module
- one fixture manifest schema plus typed read/write/check helpers
- one dedicated fixture E2E runner/config lane
- one capture/check CLI surface on the existing binary
- one dedicated fixture storage layout and sanitization pass
- focused unit tests for manifest parsing, replay setup, and offline failure behavior
- selective fixture-backed E2E suites for the highest-value surfaces

That is enough for issue #87’s first milestone.
Anything larger should clear a high scope bar.

## Main risks to watch

### 1) Auth gating blocks all replay coverage

This is the first harness blocker.
If replay tests still call the normal auth flow without a deterministic fixture state, they will fail before selectors are exercised.

Planning response:

- solve auth at the replay harness boundary
- keep live login logic untouched
- add unit tests for replay auth semantics before broadening suites

### 2) Localhost-server pressure causes broad URL refactors

The issue text mentions a local mock server, but the current services hardcode LinkedIn URLs.
A server-first implementation can turn into a repo-wide base-URL rewrite very quickly.

Planning response:

- put capture/replay behind one transport adapter
- prefer the low-refactor path first
- only introduce shared base-URL plumbing if the implementation proves it is truly necessary

### 3) Inbox needs network replay, not just saved HTML

Inbox merges DOM scraping with `waitForResponse()` messaging data.
An HTML-only fixture story will miss the highest-value inbox paths.

Planning response:

- prove HAR-backed replay early on inbox
- keep network-aware coverage as a required success criterion
- avoid calling the harness “done” after only static page extraction works

### 4) Fixture scope will sprawl across locale and account state

Connections, composer, followups, and parts of feed behavior depend on the current account state. The repo also already supports multiple selector locales.

Planning response:

- record locale explicitly in the manifest
- partition captures by page type and account state
- start with a small sanctioned matrix instead of pretending one fixture set covers everything

### 5) Service workers or missing routes can silently reach the network

A replay lane that leaks to live LinkedIn is worse than no replay lane because it can fail unpredictably and undermine account safety.

Planning response:

- block service workers
- fail closed on missing routes
- test the “missing fixture aborts fast” behavior directly

### 6) CI can become slow or brittle

The repo does not provision browsers in CI today, and large fixture suites can quickly slow the feedback loop.

Planning response:

- keep the new lane explicit and separate
- start with a smaller replay suite footprint
- preserve the current fast lint/typecheck/unit-test job

### 7) Fixtures may capture sensitive data

HAR and HTML assets can include profile data, message content, headers, and cookies.

Planning response:

- keep bulk fixtures gitignored by default
- add sanitization and redaction to the capture workflow
- commit only manifest or small sanitized samples when absolutely needed

### 8) Replay success can be mistaken for live validation

Tier 1 is not the replacement for live read/write validation issues.

Planning response:

- keep docs and CI naming explicit about replay vs live
- avoid removing or diluting the live lane
- do not claim live write safety based on replay alone

### 9) Scope creep into full parity too early

It is tempting to cover every module, CLI contract, MCP contract, and every confirm flow immediately.

Planning response:

- prioritize first proof points
- keep CLI capture/check useful but thin
- add breadth only after the harness seam is stable

## Guardrails

### Do touch

- new fixture replay helpers, manifest helpers, and runner/config files
- `setup.ts` / `helpers.ts` / related E2E harness tests
- CLI fixture commands on the existing binary
- fixture storage layout, gitignore rules, and CI wiring
- replay suites for prioritized modules

### Do not touch unless the implementation proves it is necessary

- broad `https://www.linkedin.com/...` URL rewrites across feature modules
- `headlessLogin()` / live login flows in `packages/core/src/auth/session.ts`
- keepalive/cookie-transplant scripts and live session tooling
- `packages/mcp/src/bin/linkedin-buddy-mcp.ts` and `packages/mcp/src/index.ts`
- `packages/core/src/twoPhaseCommit.ts`
- a full migration to `@playwright/test` or other new test frameworks
- heavy new server/framework dependencies before the replay seam works

### Avoid these traps

- do not overload `npm run test:e2e` with both live and replay semantics
- do not allow silent network pass-through to real LinkedIn
- do not commit a large raw fixture corpus by default
- do not encode English-only or one-account-state DOM as the universal truth
- do not call the feature done before inbox proves network replay
- do not start with a large localhost-origin refactor before the shared replay adapter works
- do not present replayed confirm paths as equivalent to live write validation

## Conventions to follow

- keep TypeScript strict and additive
- use ESM imports with `.js` extensions
- use `node:` imports for built-ins
- use type-only imports where appropriate
- keep Vitest as the test runner unless there is a compelling blocker
- keep the CLI thin; put browser fixture logic in core/test harness helpers
- preserve structured logging and artifact capture
- prefer small, typed helpers over a broad mock framework
- keep live and replay lanes explicitly separate

## Implementation compass

If the builder stays within a new replay lane, a small manifest-backed fixture system, one replay context/auth seam, prioritized suites, thin CLI capture/check commands, and explicit CI wiring, the implementation is probably on track.

If the builder starts rewriting every feature URL, mixing live and replay semantics into one runner, moving the repo to a new test framework, or chasing total module parity before inbox and selector-audit proof points are stable, the work has probably drifted out of scope.
