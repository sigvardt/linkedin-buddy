# Research brief — issue #199 Phase 0: anti-bot evasion countermeasure survey and priority ranking

Related issues: #199, #132

## Executive summary

The current repo already has a credible **typing realism layer**, but only a
lightweight **mouse / scroll / idle layer** and effectively no explicit
**fingerprint hardening** or **network-stealth shaping**.

In practice, modern anti-bot systems do not rely on one signal such as
`navigator.webdriver`. Public documentation from MDN, Cloudflare, DataDome, and
Fingerprint points to a layered model that combines:

- JavaScript/browser signals such as `navigator.webdriver`, CDP/DevTools side
  effects, and cross-surface fingerprint consistency
- interaction signals such as mouse trajectories, scroll rhythm, dwell time,
  and edit behavior
- request/network signals such as TLS / HTTP fingerprints, request sequences,
  cookie continuity, and IP reputation
- challenge and trap systems such as CAPTCHAs, hidden honeypots, decoy flows,
  and sequence anomalies

For this codebase, the highest-value next steps are **not** “deep spoof
everything.” The best first wave is:

1. extend `packages/core/src/humanize.ts` so mouse and scroll behavior are as
   rich as typing behavior
2. add a session-level rhythm layer around dwell time, warmup, idle periods,
   and navigation pacing
3. harden browser-context consistency in the context/profile layer before
   attempting fragile deep-fingerprint spoofing
4. keep challenge / honeypot detection fail-closed and add hidden-element/trap
   avoidance

The main meta-recommendation is: **prefer consistency over randomness**. Public
anti-bot material repeatedly emphasizes that mismatched signals are suspicious,
so “randomize everything” is a weaker strategy than “present a stable,
coherent, human-looking session.”

## Scope and caveats

- This brief is research-only. It proposes priorities for later phases and does
  not change runtime behavior.
- The focus is authenticated LinkedIn automation through Playwright-driven
  Chrome / Chromium.
- Public LinkedIn-specific detection details are limited. The strongest public
  evidence is LinkedIn policy/legal material plus the repo’s own observed
  checkpoint/captcha/rate-limit handling.
- Some signals, especially TLS / JA3 / JA4 and IP reputation, live partly
  outside the Playwright layer. They matter, but they are not the easiest first
  implementation targets in this repository.

## Technique inventory — what real anti-bot systems check

### 1) Mouse movement patterns

Real anti-bot systems commonly look for mouse behavior that is too straight,
too clean, too fast, or too perfectly targeted. Public prior art and vendor
material repeatedly mention:

- curved movement instead of single straight-line interpolation
- variable velocity rather than fixed-speed travel
- micro-jitter during travel or hover
- overshoot / correction before a click
- hover hesitation before interacting with a target
- missed targets, re-approach, and small corrective moves
- idle drift that keeps the pointer from looking frozen for entire sessions

Why it matters here:

- Playwright’s default pointer movement is deterministic unless the caller adds
  path shaping.
- Fingerprint and academic mouse-dynamics work both treat cursor trajectories as
  a strong behavioral input.
- Mouse realism is especially relevant for authenticated web apps where actions
  happen in dense UI regions with repeated click patterns.

### 2) Scroll behavior

Public anti-bot material also highlights scroll behavior that lacks the rhythm
of real reading. Signals include:

- variable scroll chunk size rather than constant deltas
- acceleration/deceleration and inertia-like movement
- scroll-then-read dwell periods
- occasional reverse scrolls to re-read content
- end-of-scroll hesitation before the next action
- touchpad/touch-like momentum or wheel-like event clustering

Why it matters here:

- Feed, notifications, jobs, search results, and inbox-style views naturally
  generate scroll-heavy sessions.
- A single `scrollBy({ behavior: "smooth" })` is better than an instant jump,
  but it is still far from organic browsing.

### 3) Timing and behavioral signals

This category is broader than typing speed. Real systems can combine many small
signals into a session-level classifier:

- dwell time before interacting after navigation
- read time between scrolls and clicks
- burstiness vs. perfectly regular pacing
- session warmup before mutations
- focus / blur and visibility changes
- viewport resize or small layout adjustments
- random idle periods and longer “thinking” gaps
- cross-page navigation cadence and request sequencing

Why it matters here:

- Even good mouse/scroll primitives can still look robotic when every page gets
  the same short delay pattern.
- Authenticated sessions are especially sensitive to “login then immediately do
  many high-intent actions” patterns.

### 4) Browser fingerprint hardening

This remains one of the most obvious technical surfaces. Public references point
to at least these checks:

- `navigator.webdriver`
- CDP / DevTools side effects
- old/new headless Chrome differences
- `chrome.*` surface quirks and automation artifacts
- `navigator.languages`, locale, timezone, and region coherence
- viewport/screen/platform/device metadata consistency
- plugin / MIME type surface and permissions state
- canvas / WebGL / audio / font fingerprint stability
- cross-surface consistency rather than any single property alone

Why it matters here:

- Playwright sessions can be identified by obvious browser-automation surfaces if
  the context is not hardened.
- Inconsistent combinations such as timezone, locale, IP region, and language
  can look suspicious even if each value is plausible by itself.

### 5) Network-level and request-level signals

Server-side systems can use network and sequence signals the browser page never
sees directly:

- TLS / JA3 / JA4 style fingerprints
- HTTP request fingerprints and header patterns
- IP / ASN / geolocation reputation
- cookie continuity and “fresh session every time” patterns
- navigation / request sequence analysis
- referrer/origin provenance and browsing graph coherence
- timing jitter between requests and subresource bursts

Why it matters here:

- Some of these are stronger than client-side evasion because they are harder to
  spoof from JavaScript.
- They matter most in authenticated, repeated-use workflows like LinkedIn.

### 6) Additional techniques

Other widely discussed anti-bot controls include:

- hidden honeypot elements or decoy paths
- invisible or low-friction challenge pages
- CAPTCHA escalation after suspicious sequences
- rate-pattern analysis rather than only per-minute hard limits
- sequence models that score the order of actions across a session

Why it matters here:

- The safest behavior is often to detect-and-stop early rather than keep trying.
- Hidden-element avoidance is a cheap defensive improvement even before more
  advanced behavior modeling lands.

## LinkedIn-specific considerations

Public LinkedIn documentation does not disclose a concrete detection model, but
two things are clear:

1. LinkedIn explicitly prohibits most unauthorized automation and says it uses
   proactive technical measures against scraping and similar abuse.
2. LinkedIn’s approved crawling terms require accurate identification,
   including truthful IP and user-agent information, which implies LinkedIn does
   care about request provenance and identity consistency.

Combined with the repo’s own handling of `/checkpoint`, captcha, and
`challenge_global_internal_error`, the conservative inference is:

- LinkedIn likely combines standard browser and network detection surfaces with
  account/session-level reputation and behavioral sequencing.
- For this project, a believable authenticated session is likely more important
  than one-off page tricks.
- A fail-closed design on challenge/captcha remains necessary even if more
  evasion work is added later.

## Current repo coverage — `humanize.ts` and adjacent flows

### What is already strong

The typing layer is much more mature than the rest of the anti-bot surface:

- `packages/core/src/humanize.ts:283` ships multiple typing profiles with typo
  rate, pause ranges, correction timing, and `Shift` handling.
- `packages/core/src/humanize.ts:1064` and
  `packages/core/src/humanize.ts:1084` use an in-repo QWERTY adjacency map to
  generate nearby-key typos.
- `packages/core/src/humanize.ts:1255` makes typing contextual rather than flat:
  word boundaries, repeated characters, punctuation, thinking pauses, and
  longer pauses all change cadence.
- `packages/core/src/humanize.ts:1491`,
  `packages/core/src/humanize.ts:1569`, and
  `packages/core/src/humanize.ts:1590` harden the typing path with timeout
  budgets, stuck-modifier cleanup, and safe degradation to direct input.

Operationally, that typing layer is already used in auth flows for credential
entry at `packages/core/src/auth/session.ts:248`.

### What exists but is still lightweight

Mouse, scroll, and generic timing helpers exist, but they are small wrappers:

- `packages/core/src/humanize.ts:1202` scrolls once with a random distance and a
  fixed settle pause.
- `packages/core/src/humanize.ts:1217` moves the pointer once near a target with
  a tiny offset and a small number of `steps`.
- `packages/core/src/humanize.ts:1232` optionally moves to a jittered point
  inside the target box before clicking.
- `packages/core/src/humanize.ts:1456` implements idle as a short random wait.

The keepalive flow uses those lighter helpers, not the richer typing model:

- `packages/core/src/keepAlive.ts:1683` performs a warmup idle.
- `packages/core/src/keepAlive.ts:1712` creates a fast `humanize()` wrapper.
- `packages/core/src/keepAlive.ts:1725` may perform one downward scroll, then
  idle.

### What is missing today

The following anti-bot areas are effectively absent from the main runtime:

- no Bézier / curved mouse paths
- no overshoot / correction / hover hesitation model
- no scroll-up, read-then-scroll, or inertia-like scroll rhythm
- no focus / blur / visibility / resize session signals
- no browser-context hardening for `navigator.webdriver`, plugin surfaces,
  locale/timezone/user-agent coherence, or canvas/WebGL
- no network request shaping such as provenance-aware sequencing or request
  jitter in the browser automation path
- no honeypot-aware hidden-element avoidance beyond normal selector choices

### Existing challenge handling

The repo does already detect and stop on challenges, which is the right base
policy for now:

- `packages/core/src/auth/sessionInspection.ts:48` detects rate-limit challenge
  URLs.
- `packages/core/src/auth/session.ts:315` looks for captcha/recaptcha markers.
- `packages/core/src/writeValidationRuntime.ts:201` maps checkpoint states to
  structured auth/challenge errors.

This is not evasion, but it is useful safety infrastructure.

### One documentation note

`docs/human-typing-architecture.md` is now stale relative to the implementation.
It still describes missing typo/correction and richer pause behavior, but those
capabilities now exist in `packages/core/src/humanize.ts:1255`,
`packages/core/src/humanize.ts:1792`, and
`packages/core/src/humanize.ts:1838`.

## Priority ranking

Risk scale:

- `5` = very high detection relevance
- `1` = low detection relevance

Effort scale:

- `5` = high complexity / fragile / invasive
- `1` = relatively self-contained

| 1 | Browser-context consistency baseline | locale/timezone/language/user-agent/viewport coherence, persistent-profile hygiene, obvious automation flags | 5 | 3 | High signal value, touches multiple detections at once, and fits the existing profile/context architecture better than deep spoofing. |
| 2 | Mouse trajectory realism | curved paths, variable velocity, overshoot/correct, hover hesitation, micro-jitter | 4 | 3 | The repo already has a `HumanizedPage` abstraction, so this is an obvious extension point with high behavioral payoff. |

## Recommended implementation order

### Implement first

1. **Context consistency baseline**
   - Add one explicit place where locale, timezone, user-agent, viewport, and
     related context defaults are chosen coherently.
   - Keep persistent profile continuity as the default, because stable cookies
     and stable browser state are more believable than constant fresh starts.
   - Treat `navigator.webdriver` and similarly obvious flags as baseline
     hardening, not the entire strategy.

2. **Mouse + click path upgrade inside `humanize.ts`**
   - Add curved multi-segment movement.
   - Add overshoot/correct on longer moves.
   - Add hover dwell before a click.
   - Keep this deterministic enough to test.

3. **Scroll rhythm upgrade inside `humanize.ts`**
   - Replace one-shot scrolls with chunked scroll/read loops.
   - Model occasional reverse scrolls and post-scroll dwell.
   - Prefer viewport-relative targeting over arbitrary fixed pixel ranges.

4. **Hidden-element / challenge guards**
   - Fail closed on suspicious challenge markers.
   - Avoid interacting with hidden, offscreen, zero-size, or decoy-like
     elements unless explicitly allowed.

### Implement second

5. **Session-level rhythm orchestration**
   - Introduce a higher-level behavioral policy that governs post-login warmup,
     inter-page dwell, and longer idle windows.
   - Use it across keepalive, auth, and eventual write flows so pacing is not
     fragmented.

6. **Navigation/request sequencing guardrails**
   - Preserve plausible provenance between pages.
   - Avoid abrupt “teleport” workflows that jump straight into high-value pages
     or mutations without normal browsing transitions.
   - Keep cookies and session reuse stable.

### Defer until there is measurement

7. **Deep fingerprint spoofing**
   - Canvas/WebGL/audio/plugin spoofing should be feature-flagged and measured,
     not added blindly.
   - Overly synthetic spoofing can create contradictions that are worse than the
     original baseline.

8. **Network/TLS shaping**
   - This is important, but it likely belongs in proxy/browser distribution or
     infrastructure work, not only inside repo-local Playwright code.

## Why these should be the first techniques to implement

The repo already has the right abstraction boundary for interaction realism:
`HumanizedPage`. That makes mouse and scroll improvements cheaper and safer than
inventing a whole stealth stack first.

By contrast, deep fingerprint spoofing is both high-risk and high-fragility:

- it tends to be incomplete unless maintained continuously
- it can create cross-surface inconsistencies
- it is harder to validate with unit tests than path/timing behavior
- it is more likely to break on browser upgrades

That is why the most practical first phase is:

- **baseline context consistency**
- **richer observable behavior**
- **session pacing**
- **safe aborts on traps/challenges**

## Concrete gaps to carry into future issues

These are the clearest repo-local gaps surfaced by this survey:

1. `packages/core/src/humanize.ts` needs richer mouse trajectories.
2. `packages/core/src/humanize.ts` needs richer scroll/read behavior.
3. The runtime needs a session-level pacing policy beyond local helper delays.
4. The browser/profile layer needs explicit context-consistency defaults.
5. Challenge handling is present, but trap avoidance and hidden-element guards
   are still missing.
6. Network-level shaping should be treated as a later, partly out-of-repo
   concern.

## Bottom line

This repo is already ahead of many automation projects on **typing realism**,
but still early on **mouse realism**, **scroll realism**, **session-level
behavior**, **browser-context consistency**, and **network/fingerprint
hardening**.

If the goal is the highest-value Phase 1 implementation, the best order is:

1. context consistency baseline
2. richer mouse paths
3. richer scroll/read behavior
4. session-level pacing
5. trap/challenge guards
6. only then deeper fingerprint and network work
