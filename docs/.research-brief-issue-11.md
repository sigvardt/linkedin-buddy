# Research brief — issue #11 E2E test coverage hardening

## Executive summary

Issue #11 asks for broad real-session E2E coverage: every CLI command and MCP
tool, the key two-phase commit flows, important error paths, safer operator
documentation, and a CI-compatible runner.

The repo already has a solid starting point for that work:

- a separate live-browser Vitest lane via `npm run test:e2e`
- a shared runtime helper in `packages/core/src/__tests__/e2e/setup.ts`
- read-only smoke coverage for several core services
- some real outbound two-phase commit tests for messages, comments, and posts

The main gaps are not in basic Playwright wiring. They are in coverage shape and
execution policy:

- current E2E tests exercise `packages/core` directly, not the CLI binary or MCP
  stdio server
- most mutation flows required by issue #11 are still untested
- error paths such as expired tokens, rate limits, and selector drift do not
  have E2E coverage
- the suite currently "passes" by early-returning when CDP or authentication is
  missing instead of reporting explicit skips
- write tests are not consistently opt-in: inbox and feed write tests execute
  automatically whenever a live authenticated session is present

## What exists today

### 1) Unit and E2E lanes are already separated

The root `package.json` already defines separate unit and E2E entry points:

- `npm test` runs `vitest run`
- `npm run test:e2e` runs `LINKEDIN_E2E=1 vitest run -c vitest.config.e2e.ts`

The two Vitest configs reinforce that split:

- `vitest.config.ts` includes normal `*.test.ts` files and explicitly excludes
  `*.e2e.test.ts`
- `vitest.config.e2e.ts` includes only
  `packages/core/src/__tests__/e2e/**/*.e2e.test.ts`

That means the repo already has a clean structural place for real-session tests,
but the operator surfaces required by issue #11 have not been added to it yet.

### 2) The current E2E suite is core-runtime-first

The live suite is built around `createCoreRuntime()` in
`packages/core/src/__tests__/e2e/setup.ts`.

That helper:

- reads `LINKEDIN_CDP_URL` with a default of `http://localhost:18800`
- creates one shared runtime instance for the suite
- probes CDP availability via `fetch(${CDP_URL}/json/version)`
- probes authentication via `runtime.auth.status()`

Every current E2E test file uses those helpers and then calls `runtime.*`
methods directly. No current E2E test invokes:

- the `linkedin` CLI binary in `packages/cli/src/bin/linkedin.ts`
- the `linkedin-buddy-mcp` server in `packages/mcp/src/bin/linkedin-buddy-mcp.ts`

So the current suite validates core behavior, but not command parsing, CLI JSON
shape, exit codes, MCP tool registration, stdio wiring, or MCP payload shape.

### 3) The suite already shows three E2E patterns

#### Read-only smoke tests

These current files cover live read-only paths:

- `packages/core/src/__tests__/e2e/auth.e2e.test.ts`
- `packages/core/src/__tests__/e2e/health.e2e.test.ts`
- `packages/core/src/__tests__/e2e/profile.e2e.test.ts`
- `packages/core/src/__tests__/e2e/search.e2e.test.ts`
- `packages/core/src/__tests__/e2e/jobs.e2e.test.ts`
- `packages/core/src/__tests__/e2e/notifications.e2e.test.ts`
- `packages/core/src/__tests__/e2e/connections.e2e.test.ts`
- `packages/core/src/__tests__/e2e/feed.e2e.test.ts`
- `packages/core/src/__tests__/e2e/inbox.e2e.test.ts`

#### Real write / two-phase commit tests

These files cover actual outbound prepare → confirm flows:

- `packages/core/src/__tests__/e2e/inbox-write.e2e.test.ts`
- `packages/core/src/__tests__/e2e/feed-write.e2e.test.ts`
- `packages/core/src/__tests__/e2e/post-write.e2e.test.ts`

Coverage is uneven, though:

- `post-write.e2e.test.ts` is explicitly opt-in via
  `LINKEDIN_ENABLE_POST_WRITE_E2E=1`
- `inbox-write.e2e.test.ts` and `feed-write.e2e.test.ts` do not have a similar
  opt-in gate; they run automatically whenever CDP and authentication are both
  available

That makes the current default E2E lane unsafe as a general-purpose CI or
contributor workflow.

#### Ad hoc integration helper

There is also a separate read-only helper script at `scripts/integration-test.ts`.
It connects over CDP, opens real LinkedIn pages, performs smoke checks, and
writes a Markdown report to `/tmp/linkedin-integration-results.md`.

This is useful operationally, but it is not currently:

- wired into `package.json` scripts
- part of Vitest
- part of `.github/workflows/ci.yml`

### 4) Current skip behavior is implicit, not explicit

All current E2E suites follow the same pattern:

- `beforeAll()` sets `cdpOk` from `checkCdpAvailable()`
- if CDP works, `beforeAll()` sets `authOk` from `checkAuthenticated()`
- each `it(...)` begins with `if (!cdpOk || !authOk) return;`

This prevents noisy failures on machines without a live LinkedIn session, but it
also means:

- the tests do not report as skipped
- missing-session runs can look fully green even though no real assertions ran
- CI cannot distinguish "E2E genuinely passed" from "E2E never executed"

For issue #11's CI-compatible runner requirement, this is the most important
infrastructure gap.

### 5) CI currently does not execute E2E at all

`.github/workflows/ci.yml` currently:

- runs on `ubuntu-latest`
- sets `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1`
- runs `npm ci`
- runs `npm run lint`
- runs `npm run typecheck`
- runs `npm test`

Because `npm test` excludes `*.e2e.test.ts`, the current GitHub Actions job has
no E2E coverage lane.

This is consistent with the current hosted-CI environment, which does not have:

- a provisioned authenticated LinkedIn browser session
- a configured remote CDP endpoint
- a browser download step for local Playwright launches

### CLI coverage gap

Issue #11 explicitly asks for every CLI command to have a corresponding E2E
test. That work is almost entirely still open.

The current live suite does **not** spawn or validate the CLI surfaces in
`packages/cli/src/bin/linkedin.ts`, including:

- `status`
- `rate-limit`
- `keepalive start|status|stop`
- `login`
- `search`
- `inbox list|show|prepare-reply`
- `connections list|pending|invite|accept|withdraw`
- `followups list|prepare`
- `feed list|view|like|comment`
- `post prepare|confirm`
- `actions confirm`
- `notifications list`
- `jobs search|view`
- `profile view`
- `audit selectors`
- `health`

Some of the underlying behavior is indirectly exercised through `packages/core`,
but there is no live validation today for:

- command parsing and argument validation
- JSON rendering shape
- exit-code behavior
- the relationship between `post confirm` and `actions confirm`
- CLI-specific operator flows such as `audit selectors`, `rate-limit`, and the
  daemon lifecycle commands

### MCP coverage gap

Issue #11 also explicitly asks for every MCP tool to have corresponding E2E
coverage. That work is also still open.

The repo exports 23 MCP tool constants from `packages/mcp/src/index.ts`, but no
current E2E test starts the MCP server or exercises its stdio tool contract.

The missing live MCP coverage includes these registered tool families:

- session: status, open login
- inbox: list threads, get thread, prepare reply
- profile: view
- search
- connections: list, pending, invite, accept, withdraw
- network: prepare follow-up after accept
- feed: list, view post, like, comment
- posts: prepare create
- notifications: list
- jobs: search, view
- actions: confirm

There is also one inventory mismatch worth resolving during issue #11 planning:

- `packages/mcp/src/index.ts` exports `LINKEDIN_SESSION_HEALTH_TOOL`
- `packages/mcp/src/bin/linkedin-buddy-mcp.ts` does not currently register a
  matching health tool handler

That means the project should decide whether session health is meant to be a
real MCP surface before promising one-to-one E2E coverage for it.

## Error-path coverage gaps

Issue #11 calls out three E2E error-path categories: expired tokens, rate
limits, and UI change detection.

### 1) Expired token coverage is missing from E2E

The current live suite prepares real actions and confirms them immediately. It
does not currently validate expired confirmation tokens in the live environment.

There is some low-level token coverage in `packages/core/test/twoPhaseCommit.test.ts`,
but it only covers token generation, storage, expiry math, and successful
confirmation — not a live expired-token workflow.

### 2) Rate-limit coverage is preview-only, not confirm-path E2E

The write E2E tests do verify that prepare previews include `rate_limit`
metadata, but they do not intentionally drive confirm-time `RATE_LIMITED`
behavior for:

- messages
- comments
- likes
- connection actions
- follow-up messages
- posts

So the repo currently verifies that rate-limit metadata is surfaced, not that
rate-limit failures are handled correctly in real confirm flows.

### 3) UI change detection is not exercised as an E2E assertion path

The repo has robust selector-drift machinery and a selector audit command, but
issue #11's requested "UI change detection" coverage is not yet represented as a
real E2E failure path.

Today:

- selector-heavy modules can throw `UI_CHANGED_SELECTOR_FAILED`
- selector audit exists and has unit coverage in
  `packages/core/src/__tests__/selectorAudit.test.ts`
- live E2E suites do not intentionally assert selector-drift handling or
  artifact expectations

That means the project currently has selector observability, but not the E2E
failure assertions issue #11 asks for.

## Safety rules and test-state constraints

### 1) Safe-target guidance already exists, but it is split across files

Current safety guidance is distributed across:

- `AGENTS.md`
- the README E2E section in `README.md`
- inline comments in the write E2E files
- the read-only warning block in `scripts/integration-test.ts`

The current documented rules are broadly:

- messages may only target Simon Miller unless explicitly approved
- connection requests may only target Simon Miller unless explicitly approved
- comments, likes, posts, and other public actions require explicit approval
- read-only operations are unrestricted

That is a strong safety baseline, but issue #11 will need one consolidated E2E
operator document so the runner, tests, and documentation all agree.

### 2) Connection-flow coverage needs controlled fixture state

The hardest issue #11 gap is not code coverage plumbing. It is reproducible,
safe LinkedIn state.

To fully cover the requested connection flows, the suite needs predictable test
fixtures:

- `invite` needs a target that is safe to invite and not already connected
- `accept` needs a pending inbound invitation waiting in the account
- `withdraw` needs a pending outbound invitation to a safe target
- accepted-connection follow-ups need a recently accepted sent invitation plus
  stable DB state

The current safe-target policy names Simon Miller as the allowed target for
messages and connection requests, but that alone is not enough to guarantee all
four connection states.

So issue #11 likely needs one of these before broad connection E2E can be
completed safely:

- a dedicated controlled LinkedIn fixture account strategy
- explicit approval for additional safe test accounts
- or a narrower acceptance definition for which connection flows must be live
  versus simulated

### 3) Public-action flows must stay opt-in

The current suite already proves the repo can drive real public actions, but it
also shows why those flows cannot be part of a default CI lane.

Public or semi-public actions include:

- feed likes
- feed comments
- posts

Those tests should remain explicitly opt-in, even after issue #11, because the
approval model in `AGENTS.md` is intentionally stricter than read-only smoke
testing.

## Research conclusions

1. The repo already has the right basic primitives for real-session E2E:
   separate Vitest configs, a reusable runtime helper, and proven live-browser
   flows.
2. The biggest missing piece is an explicit E2E runner contract: the current
   suite silently no-ops instead of reporting real skips, and write tests are not
   consistently opt-in.
3. Issue #11 should be treated as three related projects, not one flat test add:
   core-runtime gap filling, CLI process coverage, and MCP stdio coverage.
4. Connection hardening is blocked more by safe reproducible fixture state than
   by Playwright mechanics.
5. CI compatibility should mean: "safe to run everywhere, explicit about what
   executed, and able to skip when no authenticated session is available" — not
   "attempt every real LinkedIn mutation on every CI run."
