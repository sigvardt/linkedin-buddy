# Execution plan — issue #10 draft quality evaluation harness

## Intent

Implement issue #10 as an **offline evaluation workflow** that scores draft replies against expected attributes instead of exact reference text.
The first implementation should be able to run on synthetic repo fixtures or externally supplied candidate drafts, emit pass/fail per attribute, and stay independent from the LinkedIn/browser runtime.

The main takeaway from `docs/.research-brief-issue-10.md` is:

- keep the harness repo-native
- start provider-agnostic
- accept candidate drafts as input first
- use deterministic checks for hard constraints
- leave room for judge-based scoring where subjectivity matters

## Recommended approach

### 1) Keep the evaluator pure and outside the runtime graph

The scoring engine should live in `packages/core`, consume plain JSON inputs, and return plain JSON reports.
It should **not** depend on:

- `createCoreRuntime()`
- `profileManager`, `auth`, or `connectionPool`
- `twoPhaseCommit`, `rateLimiter`, or `db`
- any browser/CDP/session state

That keeps runs deterministic, fast, and CI-friendly.
It also means issue #10 can ship without touching the LinkedIn automation modules.

Suggested shape:

- `packages/core/src/draftQualityTypes.ts`
- `packages/core/src/draftQualityEval.ts`
- optional `packages/core/src/draftQualityJudge.ts` if the subjective-judge seam needs its own small file

### 2) Make the dataset case-first, synthetic, and validator-friendly

The harness should center on **cases** that describe thread context plus evaluation expectations.
Each case should be rich enough to support both manual review and automated scoring.

Recommended case shape:

- stable `id`
- `channel` / scenario metadata
- `thread.participants` and ordered `thread.messages`
- `expectations` for tone, length, required points, forbidden phrases, and optional notes
- optional embedded `candidateDrafts`
- `metadata` for fixture provenance, difficulty, or tags

For committed fixtures, JSON is still the best fit for this repo.
It is already natural for TypeScript, works well with nested thread data, and avoids adding another parser/configuration layer.

The first implementation should support two input patterns:

- dataset file with embedded `candidateDrafts`
- dataset file plus a separate candidates file keyed by `caseId` / `draftId`

That is enough to satisfy “manual and automated evaluation” without forcing draft generation into the MVP.

Because the repo does not currently carry a schema-validation dependency, the safest first pass is:

- TypeScript types for the dataset/report shapes
- explicit parse-and-validate helpers from `unknown`
- focused error messages for malformed fixtures

Do **not** add `zod`, `ajv`, or similar unless the builder finds the custom guards are clearly too brittle.

### 3) Split metric logic into hard checks and soft judgments

Issue #10 names three required metrics: relevance, tone match, and length compliance.
They should not all be treated the same.

Recommended split:

- **length** → fully deterministic bounds
- **relevance** → deterministic required-point coverage first, optional subjective judge later
- **tone** → deterministic forbidden-signal checks plus subjective judge/manual review for nuance

The important rule is that **hard failures should fail the draft even if an aggregate score looks good**.
The report should never make a missing required point disappear behind a nice overall average.

A good per-metric result shape can support:

- `passed`
- `score` (optional when a metric is binary)
- `mode` such as `deterministic`, `judge`, or `hybrid`
- `details` / `rationale`

That gives the builder room to grow from pure assertions into hybrid scoring without redesigning the report format.

### 4) Design subjective scoring as a plug-in, not a product commitment

The plan should leave room for **LLM-based subjective scoring** without coupling the repo to one provider.

Suggested seam:

- define a narrow `DraftQualityJudge` interface that receives a normalized case + draft
- have it return rubric-aligned judgments for subjective dimensions such as tone or overall relevance
- keep the base evaluator usable with no judge configured

That means the subjective path should be **LLM-shaped**, but the smoke path should still work with:

- deterministic checks only
- heuristic/manual review placeholders
- later provider-backed adapters

Avoid:

- hard-coding model names into dataset or report types
- requiring network access for the base CI path
- making judge output the only source of pass/fail

### 5) Prefer a thin CLI entrypoint over a scripts-only MVP

This is the main repo-specific caveat to the research brief’s suggested `scripts/evaluate-draft-quality.ts` runner.

Today, `npm run typecheck` and `npm run build` only cover the package references in `tsconfig.json`.
The `scripts/` directory sits outside that graph.

Planning response:

- keep the evaluator in `packages/core` no matter what
- prefer a thin command in `packages/cli` for the supported operator entrypoint
- only add a standalone `scripts/evaluate-draft-quality.ts` helper as a convenience wrapper, not as the only supported surface, unless the TypeScript build graph is deliberately expanded

Possible command shapes:

- `linkedin audit draft-quality --dataset <path> [--candidates <path>] [--json] [--output <path>]`
- `linkedin eval drafts --dataset <path> [--candidates <path>] [--json] [--output <path>]`

The namespace matters less than keeping the command:

- read-only
- thin
- file-based
- reusable from automation

If the builder wants to stay close to the current CLI structure, `audit` is the nearest fit.

### 6) Emit both JSON and human-readable reports

The output should serve two audiences at once:

- automation / CI
- humans inspecting failures locally

Recommended JSON report content:

- `run_id`
- `dataset_path`
- optional `candidates_path`
- summary counts and pass rate
- per-metric averages where meaningful
- per-case / per-draft results
- warnings, parse errors, or skipped items

Recommended per-draft details:

- `case_id`
- `draft_id`
- `overall.passed`
- `overall.score` when relevant
- metric results for relevance, tone, and length
- explicit missing required points / triggered forbidden signals
- judge rationale when a subjective scorer is active

The human-readable summary should mirror the repo’s selector-audit pattern:

- overall outcome
- dataset path
- pass/fail counts
- failing drafts with explicit reasons
- optional verbose per-draft breakdown

Exit semantics should stay simple:

- nonzero on malformed input
- nonzero when required checks fail in strict mode
- zero on clean pass

Thresholded or advisory-only modes can come later.

### 7) Build around Mode A first, but keep Mode B possible

Issue #10 needs both manual and automated evaluation.
The clean sequencing is:

- **Mode A** — evaluate provided candidate drafts
- **Mode B** — later let a generator produce drafts before evaluation

The first implementation only needs Mode A.
But its internal API should evaluate normalized `(case, draft)` units so a generator adapter can be added later without changing the dataset or report schemas.

## Suggested files to touch, in order

1. `packages/core/src/draftQualityTypes.ts`
   - Add dataset, metric result, report, and judge-interface types.
   - Keep the shapes explicit and JSON-serializable.

2. `packages/core/src/draftQualityEval.ts`
   - Add parse/validate helpers and pure scoring functions.
   - Make this the single home for deterministic evaluation and report aggregation.

3. `packages/core/src/index.ts`
   - Re-export the new evaluator and types so CLI and future tooling share one surface.

4. `packages/core/test/draftQualityEval.test.ts`
   - Add focused coverage for parsing, hard checks, aggregation, and judge integration seams.

5. `packages/core/test/fixtures/draft-quality/`
   - Add a tiny synthetic smoke dataset.
   - Optionally add a separate candidates fixture to cover split-input mode.

6. `packages/cli/src/draftQualityOutput.ts`
   - If human-readable CLI output is included, keep the formatting here instead of bloating `linkedin.ts`.
   - Follow the small, focused pattern already used for selector-audit output.

7. `packages/cli/test/draftQualityOutput.test.ts`
   - Add formatter coverage if a new output helper is introduced.

8. `packages/cli/src/bin/linkedin.ts`
   - Add the read-only command surface.
   - Keep argument parsing, JSON/human switching, and exit-code mapping thin.

9. `package.json`
   - Only if a convenience alias such as `npm run eval:draft-quality -- --dataset ...` meaningfully improves ergonomics.
   - Do not add this unless it earns its keep.

10. `scripts/evaluate-draft-quality.ts`
    - Optional convenience wrapper only.
    - If added, document clearly whether it is a thin pass-through and how it fits the current build/typecheck boundaries.

  - use `createRunId()` for report IDs
- `packages/core/src/errors.ts`
  - use `LinkedInAssistantError` for dataset validation and CLI precondition errors
- `packages/core/src/index.ts`
  - follow the existing re-export rule for new core modules
- `packages/cli/src/bin/linkedin.ts`
  - reuse the existing JSON printing and user-facing error handling patterns
- `packages/cli/src/selectorAuditOutput.ts`
  - use as the closest pattern for dual JSON + human summary output
- `packages/cli/test/selectorAuditOutput.test.ts`
  - use as the pattern for formatter coverage
- `packages/core/test/*.test.ts`
  - use this test area for pure evaluator coverage

### Reuse conceptually, but do not force shared abstractions yet

- local text-normalization helpers already repeated across feature modules
- report-summary patterns from selector audit
- small helper-centric module shapes instead of framework-heavy layers

This does **not** mean issue #10 should start by extracting a repo-wide text utility.
A local normalization helper inside the draft-quality module is perfectly reasonable for the first implementation.

## New code that is justified

The MVP will likely need:

- one new core evaluator module
- one new core type family
- one small judge abstraction or hook if subjective scoring is included
- one small synthetic fixture set
- one thin CLI command
- one small CLI formatter module if human output needs structure

That is enough for the first implementation.
Anything bigger should clear a high scope bar.

## Main risks to watch

### 1) Entry-point drift: standalone scripts are outside the current build graph

This is the most important repo-specific risk.
If the builder adds only `scripts/evaluate-draft-quality.ts`, the runner may be linted but will not be covered by the current `npm run typecheck` or `npm run build` flow.

Planning response:

- keep the supported entrypoint inside `packages/cli`, or
- intentionally expand the TypeScript project graph if scripts must be first-class

### 2) Provider coupling sneaks in through subjective scoring

It will be tempting to bind the evaluator directly to one SDK or model prompt.

Planning response:

- define a narrow judge contract first
- keep provider config outside dataset/report types
- make the base smoke path work with no external judge

### 3) Hard and soft checks get collapsed into one opaque score

A single overall number is convenient but dangerous.

Planning response:

- preserve per-attribute pass/fail in the report
- let hard misses fail independently of aggregate score
- keep rationales alongside subjective decisions

### 4) Fixture quality drifts toward real LinkedIn content

The dataset should be realistic, but committed fixtures must stay synthetic or strongly redacted.

Planning response:

- store only synthetic/redacted threads in git
- document that larger calibration sets can live outside the repo if needed

### 5) Schema churn if generator mode arrives later

If the evaluator only works with embedded drafts, later automation will create avoidable churn.

Planning response:

- support either embedded candidates or separate candidate input keyed by `caseId`
- keep the evaluator centered on normalized case/draft pairs

### 6) Tone heuristics become brittle or overconfident

Tone is the easiest metric to overstate.

Planning response:

- use heuristics as evidence, not faux certainty
- reserve subjective-judge/manual-review fields for nuance
- keep the tone vocabulary small at first

## Guardrails

### Do touch

- pure core evaluation code
- core exports
- synthetic fixtures
- CLI entrypoint / output formatting
- focused unit tests

### Do not touch unless the implementation proves it is necessary

- `packages/core/src/runtime.ts`
- `packages/core/src/twoPhaseCommit.ts`
- `packages/core/src/db/database.ts`
- `packages/core/src/profileManager.ts`
- `packages/core/src/auth/`
- `packages/core/src/linkedin*.ts` feature modules
- `packages/mcp/src/bin/linkedin-mcp.ts`
- `packages/mcp/src/index.ts`

### Avoid these traps

- do not require a LinkedIn session, browser, or CDP connection
- do not make exact-text similarity the main pass/fail gate
- do not add a provider SDK before the judge seam is defined
- do not commit real inbox or thread data
- do not hide metric-specific failures behind one “quality score”
- do not let the CLI command own business logic that belongs in core

## Conventions to follow

- keep TypeScript strict and additive
- use ESM imports with `.js` extensions
- use `node:` imports for built-ins
- use `unknown` + type guards instead of `any`
- use type-only imports where appropriate
- keep JSON shapes stable and serializable
- keep output deterministic when no judge is configured
- prefer small helper functions over framework-heavy abstractions

## Implementation compass

If the builder ends up with a pure core evaluator, a tiny synthetic dataset, focused tests, and one thin read-only entrypoint, the work is probably on track.

If the builder starts wiring browser runtime, adding provider-specific SDKs to the core schema, or treating an untyped standalone script as the only supported surface, the work has probably drifted out of scope.
